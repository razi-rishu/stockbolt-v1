IMPLEMENTATION EXPLANATION
--------------------------------------------------------------------------------
  1. NEW parameterized entitlement helper (migration, hand-applied):
       public.company_has_api_access(p_company_id uuid) RETURNS boolean
     Applies the SAME rule as the existing zero-arg version (single source of
     truth): an active/trialing OR grandfathered subscription on a plan whose
     features.api_access = true — but keyed by the ARGUMENT, not auth.uid().
     WHY a new overload: the zero-arg company_has_api_access() resolves the
     company via current_user_company_id() = profiles WHERE id = auth.uid(); the
     Edge Function authenticates by API key (service_role, no user JWT), so
     auth.uid() is NULL and the zero-arg helper would return false for EVERYONE.
     SECURITY: SECURITY DEFINER, search_path pinned; EXECUTE revoked from
     PUBLIC/anon/authenticated and granted ONLY to service_role (it takes an
     arbitrary company_id and must not let a tenant probe another's entitlement).
     The zero-arg function is left untouched.

  2. authenticate() (supabase/functions/api/index.ts) — after the key is resolved
     and its revoked/expired checks pass, it now calls:
        supabase.rpc('company_has_api_access', { p_company_id: data.company_id })
     and:
        - on rpc error   -> 500 internal ("Entitlement check failed.")
        - if not entitled -> 402 payment_required
                             ("API access is not included in this company's
                              current plan.")
        - else            -> returns the ApiKey as before.
     This closes the per-request gap: a key minted while entitled is now rejected
     once its company's entitlement lapses (trial ends, downgrade, cancel),
     without touching or revoking the key itself.

  What was deliberately NOT done (per scope):
     * No caching — the check runs on every request (one rpc round-trip).
     * No auto-revoke / expiry of api_keys on downgrade.
     * No change to create_api_key, the Developer UI, or subscription logic.
     * No change to any app (src/) code.


WHY THIS IS SAFE / IN SCOPE
--------------------------------------------------------------------------------
  * Additive: one new SQL function + a 13-line guard in authenticate(). No
    existing object is altered (the zero-arg helper, create_api_key, UI, and
    subscription tables are untouched).
  * Verified read-only before delivery: the entitlement query returns TRUE for
    the one live key's company (so the existing integration keeps working) and
    FALSE for a non-entitled company id.
  * The only behaviour change is that a NON-entitled company's requests now get
    402 instead of 200 — exactly the intended fix.
