IMPLEMENTATION EXPLANATION
--------------------------------------------------------------------------------
  The single choke point for ALL spreadsheet imports is `readPickedFile(file)` —
  it is the only caller of `parseXLSX` (which calls the vulnerable `XLSX.read`),
  so guarding it protects the entire Excel-import surface in one place. The
  function was rewritten to validate three things and return an error result
  BEFORE any `file.text()` / `file.arrayBuffer()` / `parseXLSX` runs:

    Guard 1 — Extension allow-list (unchanged behaviour, now an explicit
      up-front reject): ext must be one of csv / txt / xlsx / xls.

    Guard 2 — Size cap (NEW): new exported constant
        export const MAX_IMPORT_BYTES = 10 * 1024 * 1024;   // 10 MB
      Rejects a file whose `size` exceeds the cap, and also rejects a 0-byte
      file. This runs before the file is read into memory, so an oversized or
      crafted workbook can no longer reach the main-thread parser (kills the
      trivially-triggerable xlsx ReDoS / memory-DoS leg). 10 MB comfortably
      covers real ERP imports (products, contacts, opening balances).

    Guard 3 — MIME validation IN ADDITION to the extension (NEW): a per-extension
      allow-list of concrete MIME types (ALLOWED_MIME). `file.type` is
      unreliable, so unknown/generic types ('' and application/octet-stream) are
      TOLERATED (fall back to the extension) and never used to reject; only a
      CONCRETE reported type that contradicts the extension is rejected
      (e.g. an .xlsx reported as text/html, or a .csv reported as
      application/zip). This is stricter than an extension-only check without
      false-rejecting legitimate uploads whose browser MIME is blank.

  Only after all three guards pass does the function read the file and parse
  (CSV via PapaParse, XLSX via parseXLSX -> XLSX.read). Rejections return the
  existing ParseResult error shape ({ rows:[], headers:[], errors:[msg] }), so
  the Import UI surfaces the message through its normal error path — no UI change
  needed.

  What was deliberately NOT touched (per scope):
    * parseXLSX / XLSX.read internals — unchanged (library swap is P1c, deferred).
    * No Web Worker (P1b, deferred).
    * Export path (stringifyXLSX / downloadXLSX / downloadCSV / XLSX.write) —
      untouched (git diff shows no export function changed).
    * No dependency / package.json / lockfile change; no `npm audit fix`.
    * No database or posting code.


WHY THIS IS SAFE / IN SCOPE
--------------------------------------------------------------------------------
  * One presentational/data-layer file; pure input validation before parsing.
  * Valid files behave exactly as before (same routing, same parse, same result).
  * Invalid files (too big, empty, wrong concrete MIME, unsupported ext) get a
    clear error and are NOT parsed — closing the easy DoS vector and adding a
    hygiene layer before the known-vulnerable XLSX.read.
  * No secrets, no network, no DB, no export behaviour affected.
