WORKFLOW EXPLANATION
--------------------------------------------------------------------------------
  name: CI
  Triggers:
    * push to `main`              — runs on every commit that lands on the deploy
                                    branch (in parallel with Vercel's deploy).
    * pull_request targeting main — runs on every PR before merge.
  Concurrency:
    * group ci-<ref>, cancel-in-progress: true — a newer push cancels the older
      run on the same branch/PR to save Actions minutes.
  Permissions:
    * contents: read — least privilege; the job only reads the checked-out code.
  Job `typecheck-build` (ubuntu-latest):
    1. actions/checkout@v4
    2. actions/setup-node@v4  (node 20, cache: npm)  — Node 20 LTS satisfies the
       repo's engines ">=18"; npm cache keyed off package-lock.json.
    3. npm ci            — clean, lockfile-exact install (package-lock.json present).
    4. npm run typecheck — `tsc --noEmit`, strict typecheck of src + tests.
    5. npm run build     — `tsc -b && vite build`, the production build.

  What it gives you:
    * An INDEPENDENT, automated gate that a change typechecks and builds — one
      that does not depend on the developer running anything locally, and that
      shows a pass/fail status on each push/PR. It can later be made a REQUIRED
      status check to block deploy (that wiring is out of this scope).

  What it deliberately does NOT do (and why):
    * NO lint step. `npm run lint` currently reports 328 pre-existing errors;
      fixing them is application-code work outside H5. Per owner decision, lint is
      omitted now and will be added as a REQUIRED step after a separate
      lint-cleanup task. (Not made "non-blocking" either — the owner did not want
      a green CI that silently ignores lint.)
    * NO test job. `npm run test:regressions` touches a live Supabase DB; running
      it in CI needs a NON-production target (Audit H4 staging), which is
      deferred. Adding it now would either hit production or need the prod secret
      — both unwanted. Hence NO secrets are referenced anywhere in the workflow.
    * NO change to Vercel / deploy config. "Deploy only on passing checks"
      (branch protection / required checks) is a separate, later step.


WHY THIS IS SAFE / IN SCOPE
--------------------------------------------------------------------------------
  * Additive: one new file under .github/workflows/. Changes no app, DB, or
    deploy behaviour on its own.
  * No secrets, no database, no external service — cannot leak or mutate anything.
  * All three steps are verified green locally before delivery (see checklist),
    so the pipeline is green on first run, not red.
