IMPLEMENTATION EXPLANATION
--------------------------------------------------------------------------------
  BROWSER (items 1 + 2):
   * src/lib/error-reporting.ts — one seam for client errors:
       - reportError(error, source, context)   — never throws.
       - defaultReporter                        — structured console.error
                                                  (keeps local dev visibility).
       - setErrorReporter(next)                 — PLUG POINT to install an
                                                  external tracker later without
                                                  touching any call site.
       - installGlobalErrorHandlers()           — idempotent; adds window 'error'
                                                  and 'unhandledrejection'
                                                  listeners, capturing the errors
                                                  React cannot see (async code,
                                                  event handlers, rejected
                                                  promises) that previously
                                                  vanished entirely.
   * src/main.tsx — calls installGlobalErrorHandlers() at startup, before render,
     so early errors are captured.
   * src/components/error-boundary.tsx — componentDidCatch now calls
     reportError(error, 'react', { componentStack }) instead of a bare
     console.error TODO. Fallback UI is unchanged.

  EDGE (items 3 + 4):
   * supabase/functions/api/index.ts — added reportEdgeError(e, context): a
     structured, stack-carrying console.error (captured by the Supabase Edge
     function logs), and the SINGLE seam where an external tracker would
     capture(e) later. The top-level handler changed from `catch (_e) { ...swallow }`
     to `catch (e) { reportEdgeError(e, {method, path, company_id}); return 500 }`
     — the exception is now reported (with its stack) rather than discarded, and
     the response is still a safe generic 500. (This also resolves the
     `'_e' is defined but never used` lint finding from H8.)

  What was deliberately NOT done (per scope):
   * No Sentry/Datadog SDK, DSN, account, or secret — no external monitor
     configured. setErrorReporter is the seam for that later.
   * No uptime monitor, no health endpoint, no DR doc, no PITR, no restore drill.
   * No business-logic or DB-schema change; api_request_log insert unchanged.


WHY THIS IS SAFE / IN SCOPE
--------------------------------------------------------------------------------
  * Additive: one new file + three small wirings; no behaviour of the accounting/
    inventory/API business paths changes.
  * The default reporter still console.errors, so nothing that worked before is
    lost; the NEW capability is capturing previously-invisible errors (client
    global handlers) and previously-DISCARDED errors (Edge catch).
  * No external service, secret, or dependency; nothing to leak or misconfigure.
  * Edge behaviour on error is unchanged from the caller's view (still a 500) —
    only the internal handling changed from "discard" to "report".


ROLLBACK PLAN
--------------------------------------------------------------------------------
  Pure code, no data/schema/dependency. Either:
    * git revert the H6-P1 commit; or
    * manually: delete src/lib/error-reporting.ts; remove the
      installGlobalErrorHandlers import+call from main.tsx; restore
      error-boundary.tsx's console.error; restore the Edge catch to `catch (_e)`
      and remove reportEdgeError.
  No production impact to undo. The Edge change only takes effect once the Edge
  Function is redeployed (see checklist), so reverting before deploy is a no-op
  on the live function.
